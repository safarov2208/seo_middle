## 2. КАК СО МНОЙ ОБЩАТЬСЯ — СТРОГИЕ ПРАВИЛА

### 2.1. Стиль ответов

- **Коротко и по делу.** Никакой воды, никаких длинных вступлений типа «Отличный вопрос! Давайте разберёмся...». Сразу к сути.
- **Чётко структурировано.** Используй заголовки, списки, блоки кода. Один шаг — один логический блок.
- **Орфографично и грамотно.** Без опечаток, без рваной пунктуации.
- **Эстетично.** Код в блоках с указанием языка (```dart, ```kotlin, ```bash, ```json). Команды терминала — отдельным блоком. Названия файлов и кнопок — `моноширинным` или **жирным**.
- **На русском языке** всегда. Технические термины (Activity, Widget, Provider, Gradle) можно оставлять английскими — это нормально.

### 2.2. Поэтапность — самое важное

- **Один ответ = один логический шаг.** Не пытайся в одном сообщении сделать «настроить проект + написать UI + настроить sing-box + добавить локализацию». Это перегруз. Разбивай на маленькие куски.
- В конце каждого ответа задавай **один контрольный вопрос**: «Готово? Видишь у себя [конкретный результат]? Если да — идём к следующему шагу.»
- **Жди моего ответа** прежде чем продолжать. Не вываливай сразу следующие 5 шагов.
- Если я говорю «следующий» / «дальше» / «готово» — переходи к следующему шагу.
- Если я говорю «ошибка», «не работает», «не вижу» — переключайся в режим отладки: спрашивай скрин, лог, точное сообщение об ошибке. Не угадывай.

### 2.3. Когда я ошибаюсь — исправляй меня

- **Никогда не говори «ты прав», если я не прав.** Если я предлагаю плохое решение, плохую архитектуру, плохое название переменной, неправильное понимание концепции — **прямо скажи**, что это неправильно, и объясни почему. Это критично — я плачу за честную экспертизу, а не за поддакивание.
- Используй формулировки типа: «Тут поправлю — на самом деле X работает не так, как ты думаешь. Правильно вот так: ...».
- Если есть **техническая невозможность** того, что я прошу — прямо скажи «это сделать нельзя по такой-то причине», предложи альтернативу.

### 2.4. Когда задавать уточняющие вопросы

- Если я даю двусмысленную задачу — задай **один-два** уточняющих вопроса, не пять.

- Если я уже что-то ранее решил в этом диалоге — **запоминай и не переспрашивай**.

### 2.7. Стиль кода — строгие правила

Каждая строка кода, которую ты мне даёшь или подсказываешь Copilot'у, должна соответствовать этим правилам:

- **Никаких inline-комментариев в коде.** Ни `// делаем то-то`, ни `// TODO`, ни `// FIXME`, ни личных подписей. Код должен быть **самообъясняющим** через хорошие имена переменных, функций и классов. Я ненавижу мусорные комментарии.
- **Doc-комментарии (`///` в Dart)** допустимы **только** для публичного API классов и функций, которые экспортируются из библиотеки. Внутри тела функций — никаких комментариев.
- **Никаких заглушек `print(...)` и `debugPrint(...)`** в production-коде. Логирование — только через единый logger (`logger` package), и только в debug-сборке.
- **Никаких магических чисел и строк.** Все константы вынесены в `lib/core/constants/`.
- **Imports отсортированы:** dart → flutter → packages → relative, между группами пустая строка. Использовать `import_sorter` или встроенный форматтер.
- **Один файл — одна ответственность.** Если файл больше 300 строк — разбиваем на несколько.
- **Именование:** классы `PascalCase`, переменные/функции `camelCase`, константы `lowerCamelCase` (Dart-стиль), приватные с префиксом `_`. Файлы `snake_case.dart`.
- **Никакого dead code.** Закомментированные блоки, неиспользуемые импорты, неиспользуемые переменные — удаляются мгновенно. `flutter analyze` должен проходить с **нулём предупреждений**.
- **Линтер:** в `analysis_options.yaml` подключаем `flutter_lints` + `very_good_analysis` (строжайший набор правил). Любое предупреждение — это ошибка, чиним сразу.
- **Форматирование:** обязательно `dart format .` перед каждым коммитом. Никаких смешанных tab/space, никаких длинных строк >80 символов (исключение — URL и длинные строковые литералы).
- **Null safety strict:** никаких `!` (force unwrap) без крайней необходимости. Используем `?.`, `??`, паттерн-матчинг, `late final` где уместно.
- **Никакого `dynamic`** без чёткой причины. Все типы явные.
- **Async/await везде**, никаких `.then().then().then()` цепочек.
- **Обработка ошибок:** все async операции в `try/catch`, ошибки превращаются в типизированные `Failure` объекты (через `freezed`) и поднимаются в UI как состояние, никаких unhandled exceptions.
- **Тесты:** для каждой бизнес-логики (репозитории, use-cases, парсеры) — unit-тест в `test/`. Минимальное покрытие критичной логики 70%.

**Применяй эти правила к каждому куску кода, который мне выдаёшь. Не давай мне грязный код «на потом почистим». Сразу чисто.**

## 4. ЦЕЛЬ ПРОЕКТА — VPNjon

Создать **бесплатное анонимное VPN-приложение для Android** (в первой фазе) и iOS (во второй фазе) под брендом **VPNjon**, для пользователей из Таджикистана. Внутри — свои сервера с протоколом VLESS + Reality + XHTTP. Пользователь скачивает приложение, выбирает язык, видит список серверов, нажимает Connect — и работает.

Никаких регистраций, никаких подписок, никакой авторизации. Полная анонимность. Сервера и ключи раздаёт администратор (то есть я) через защищённый JSON-конфиг, который приложение тянет с сервера.

**Конечные цели:**
1. Опубликовать APK в Google Play под названием **VPNjon**.
2. Иметь Telegram-бота для админки, через который я обновляю список серверов, отправляю popup-сообщения юзерам, ротирую ключи.
3. Иметь надёжный механизм защиты VLESS-ключей от выгрузки реверс-инженерами.
4. Подготовить базу для iOS-версии (Фаза 2).

|---|---|

| Хранение конфига на сервере | **JSON-файл на диске** (без БД, для простоты) |
| Шифрование payload | **AES-256-GCM**, ключ зашит в нативной библиотеке через NDK / FFI |
| Защита трафика к API | **HTTPS + Certificate Pinning + HMAC-SHA256 подпись запросов** |
| Локальное хранилище в приложении | **flutter_secure_storage** (Android Keystore) |
| State management во Flutter | **Riverpod** (более современный, чем Provider/Bloc) |
| Локализация | **flutter_localizations + intl** (RU + TJ) |
| Логотип/иконки | **Программно через Flutter `CustomPainter`** — нет файлов-картинок, всё векторно, идеально на любом разрешении. Codex рисует сам, никаких внешних AI-генераторов |

**Важно про Hiddify-Next:** мы НЕ используем его как готовое приложение. Мы **форкаем** репозиторий, **выкидываем 80% UI** (он перегружен настройками для гиков), оставляем только ядро sing-box + механизм соединения + парсер VLESS, и поверх пишем СВОЙ UI в стиле VPNjon. Это сэкономит месяцы работы по сравнению с написанием sing-box-обвязки с нуля.

## 7. ПОЛНАЯ ФУНКЦИОНАЛЬНАЯ СПЕЦИФИКАЦИЯ

### 7.1. Экраны приложения (по порядку появления у юзера)

**Экран 0 — Splash (1 секунда)**
- Логотип VPNjon по центру, лёгкая пульсация / fade-in.
- Тёмный фон, glassmorphism, blurred-circles на заднем плане.
- На фоне делаем preload: проверяем выбранный язык в local storage, проверяем кешированный config.json.

**Экран 1 — Выбор языка (только при первом запуске)**
- Заголовок: «Выберите язык / Забонро интихоб кунед».
- Две большие тапабельные карточки (glassmorphism) — «Русский 🇷🇺» и «Тоҷикӣ 🇹🇯».
- После выбора → сохраняем в `flutter_secure_storage` ключ `app_lang` → переход на онбординг.

**Экран 2 — Онбординг (3 слайда swipe, только при первом запуске)**
- Слайд 1: «Безопасность» — иконка щита, текст про защиту трафика.
- Слайд 2: «Скорость» — иконка молнии, текст про быстрые сервера.
- Слайд 3: «Свобода» — иконка глобуса, текст про доступ ко всему интернету.
- Внизу — индикатор страниц (3 точки) и кнопка «Продолжить» / «Давом» (на последнем слайде — «Начать» / «Оғоз»).
- Кнопка «Пропустить» в правом верхнем углу.

**Экран 3 — Bootstrap / Загрузка серверов (только при первом запуске или если кеш пуст)**
- Сразу после онбординга юзер попадает на этот экран. **Никаких серверов, никаких подписок, никакого VPN ещё нет.**
- По центру — большая мягкая иконка облака со стрелкой вниз (нарисованная программно через `CustomPainter`).
- Под иконкой — заголовок «Загрузить серверы» / «Серверҳоро бор кардан».
- Под заголовком — описание мелким шрифтом: «Нажмите, чтобы получить список доступных VPN-серверов» / «Барои гирифтани рӯйхати серверҳои VPN зер кунед».
- В центре экрана — **большая круглая glassmorphism-кнопка** «Загрузить» / «Бор кардан» с пульсацией.
- При тапе:
  1. Кнопка превращается в спиннер.
  2. Приложение делает запрос на `api.vpnjon.com/v1/c?...` (с HMAC-подписью).
  3. Получает зашифрованный ответ, расшифровывает AES-GCM, парсит JSON.
  4. Сохраняет в защищённое локальное хранилище.
  5. Если успех → fade-переход на главный экран с анимацией появления карусели серверов снизу.
  6. Если ошибка → красный toast «Не удалось загрузить серверы. Проверьте интернет.» и кнопка снова активна.
- Этот экран показывается **один раз при первой установке**. Дальше при запуске сразу открывается главный (с кешированными серверами), а свежий config подтягивается фоновым обновлением.
- Если юзер удалит данные приложения через настройки Android — экран Bootstrap появится снова при следующем запуске.

**Экран 4 — Главный (Home)**

Структура сверху вниз:

1. **Верхняя панель** (top bar):
   - Слева — иконка настроек (шестерёнка). Тап → экран настроек.
   - По центру — название текущего выбранного сервера и его флаг (`🇹🇯 Душанбе 1`).
   - Справа — иконка обновления (круговая стрелка). Тап → принудительно загружает свежий config с сервера, анимация вращения, toast о результате. Между тапами 10 секунд cooldown.

2. **Большая круглая кнопка Connect** (примерно 220×220 dp, по центру верхней половины экрана):
   - Стиль glassmorphism, тонкое светящееся кольцо вокруг.

3. **Блок статистики** (под кнопкой, видим только в состоянии Connected):
   - Время подключения (uptime): `00:14:23` крупным шрифтом.

   - В состоянии Disconnected этот блок скрыт или показывает заглушку «Не подключено».

### 7.2. Экран подключения / процесс Connect

- Тап на Connect → состояние «Connecting», кнопка анимируется (вращающееся кольцо).
- Запрос разрешения VPN у системы Android (диалог «Разрешить VPNjon создавать VPN-соединение?») — показывается **только при первом подключении**, потом Android запоминает.
- Параллельно: бридж в нативный код, передаём sing-box JSON-конфиг для выбранного сервера, sing-box создаёт TUN-интерфейс через `VpnService`.
- Когда соединение установлено → состояние «Connected», запускается счётчик, начинаем стримить статистику скорости из sing-box каждую секунду.
- При тапе на Connected → дисконнект, состояние «Disconnected».
- Если соединение упало (потеря интернета) → автоматический re-connect через 3 секунды, не больше 3 попыток, потом показываем toast и переходим в Disconnected.
- Все состояния хранятся в Riverpod-провайдере `connectionStateProvider`.

**Логика клиента:**

### 7.6. Бекенд — `api.vpnjon.com`

**Стек:** FastAPI + Uvicorn, под systemd на `141.98.6.133`, за nginx, за Cloudflare.

**Эндпоинт:** `GET /v1/c?t=<HMAC>&v=<client_version>&d=<device_id_hash>`

**Логика:**
1. Проверить параметр `t` — это HMAC-SHA256 от строки `<unix_hour>:<device_id_hash>:<client_version>`, вычисленный с секретом `HMAC_SECRET`. Секрет известен только серверу и приложению.
2. Если HMAC валиден для текущего часа или предыдущего часа (запас на скачок времени) — продолжаем. Иначе → `403`.
3. Применить rate limit: не больше 30 запросов в час с одного IP. Иначе → `429`.
4. Прочитать `/var/lib/vpnjon/config.json` с диска.
5. Зашифровать содержимое алгоритмом **AES-256-GCM**, ключ `AES_KEY` (32 байта, hex в .env).
6. Вернуть JSON: `{"v": 1, "iv": "<base64>", "tag": "<base64>", "data": "<base64-ciphertext>"}`.
7. Заголовки: `Cache-Control: no-store`, `ETag: <md5(version)>`. Cloudflare можно настроить кешировать ответы по этому ETag на 5 минут — снизит нагрузку.

**Эндпоинт:** `GET /v1/health` → `{"ok": true, "version": 12}` без авторизации, для мониторинга.

**Эндпоинт админский:** `POST /v1/admin/reload` — вызывается из Telegram-бота, перечитывает `config.json` с диска (если файл изменили извне), сбрасывает кеш Cloudflare через CF API. Защищён отдельным admin-токеном.

### 7.7. Telegram-бот админки (отдельный проект, делаем после приложения)

**Стек:** aiogram 3.x, Python 3.11+, SQLite (только для логов команд), запуск под systemd.

**Доступ:** только мой Telegram user_id (зашит в .env). Все остальные → «Нет доступа».

**Команды:**
- `/start` — приветствие, список команд.
- `/list` — показать все сервера из текущего `config.json`, с их `id`, `name`, `country`, `enabled`, `load`.
- `/add_server` — пошаговый диалог: спрашивает name → country (флаг подбирается автоматически по ISO-коду) → city → vless-ключ → load. После ввода всех полей — превью, кнопки «Сохранить» / «Отмена». При сохранении: бот добавляет объект в `config.json`, инкрементит `version`, обновляет `updated_at`, сохраняет файл, дёргает `/v1/admin/reload`.
- `/edit_server <id>` — выбрать поле для редактирования.
- `/delete_server <id>` — с подтверждением.
- `/toggle_server <id>` — переключить `enabled` true/false.
- `/popup` — диалог создания popup'а: title (RU) → title_tj → body (RU) → body_tj → опциональное изображение (можно загрузить файл, бот сам зальёт на сервер) → дата истечения. Превью, сохранение в `popups[]`, инкремент version, reload.
- `/popups` — список активных popup'ов с возможностью удалить.
- `/rotate_keys` — пройтись по всем серверам, для каждого вызвать Remnawave API на перегенерацию UUID, обновить ключи в `config.json`. (Эта команда сложнее, делаем отдельно после основного функционала.)
- `/stats` — показать сколько серверов активно, текущая `version` конфига, время последнего обновления.
- `/refresh_cdn` — принудительно сбросить кеш Cloudflare для `api.vpnjon.com/v1/c`.
- `/backup` — скинуть мне в чат текущий `config.json` файлом.
- `/restore` — принять `config.json` файлом и заменить текущий (с подтверждением).

**Принципы бота:**
- Все ответы на русском.
- Используем aiogram FSM для пошаговых диалогов.
- Inline-кнопки для подтверждений (Сохранить / Отмена / Удалить).
- После каждой модификации — бот пишет «✅ Сохранено. Версия конфига: 13. Кеш сброшен.»

### 7.8. Логотип, иконка приложения и графика — всё рисуется программно

**Принцип:** у меня **нет ни логотипа, ни иконки, ни картинок, ни дизайнера**. Codex генерирует всё сам, никаких внешних AI-генераторов изображений (Midjourney, DALL-E, Stable Diffusion) и никаких заказов фрилансерам. Решение — векторная графика прямо в коде через Flutter `CustomPainter` плюс `flutter_launcher_icons` для иконки в системе.

**Концепция логотипа VPNjon:**
- Минималистичный знак: стилизованный щит со стилизованной буквой **«J»** или с молнией внутри, в круглом контейнере с градиентом индиго → фиолет.
- Никаких реалистичных деталей, никаких бликов, никакой 3D-перспективы. Чистый плоский геометрический стиль, как у современных iOS-приложений 2025-2026.
- Должен хорошо выглядеть и в размере 24×24 (иконка в шапке), и в 1024×1024 (Play Store).

**Что Codex обязан сделать:**

1. **Виджет `VpnJonLogo`** в `lib/presentation/widgets/branding/vpnjon_logo.dart` — кастомный виджет на базе `CustomPainter`, рисует логотип векторно. Принимает параметр `size` (double) и опционально `color`. Используется во всех местах: Splash, top bar, About-секция настроек.

   flutter_launcher_icons:

     ios: true
     image_path: "assets/icon/icon_source.png"
     adaptive_icon_background: "#0A0E1A"
     adaptive_icon_foreground: "assets/icon/icon_foreground.png"

   Источник `icon_source.png` (1024×1024) Codex генерирует **программой** — пишет одноразовый Dart-скрипт `tools/generate_icon.dart`, который рендерит `VpnJonLogo` в `Image` и сохраняет в PNG через `image` package. Я просто запускаю `dart run tools/generate_icon.dart` и получаю готовый файл.

4. **Splash screen** — нативный Android splash через `flutter_native_splash`:
   ```yaml
   flutter_native_splash:

### 7.9. Split Tunneling — обход VPN для российских сервисов (КРИТИЧНО)

**Цель:** российские приложения (банки, маркетплейсы, госуслуги, соцсети) и российские сайты (`.ru`, `.рф`, `.su`, конкретные домены) **должны работать напрямую через домашний интернет, минуя VPN-туннель**. Российские банки часто блокируют IP-адреса зарубежных VPN, маркетплейсы могут отказываться доставлять, госуслуги блокируют вход. Юзер должен получать доступ к Telegram / YouTube / Instagram через VPN, но Сбербанк, Wildberries, Yandex Maps, Госуслуги — напрямую.

**Реализация — два уровня одновременно:**

#### Уровень 1: Per-app split tunneling (Android `VpnService` disallowed apps)

Android `VpnService.Builder` имеет метод `addDisallowedApplication(String packageName)`. Если передать в него package name приложения — трафик этого приложения **полностью** идёт мимо VPN-туннеля, как будто VPN не запущен.

Codex обязан:

1. Создать файл `lib/core/data/russian_apps_blacklist.dart` — Dart-константу со списком package name'ов всех российских приложений (полный список ниже).
2. На нативной стороне Android (Kotlin, в `VpnConnectionService.kt`) при старте VPN пройти по списку, проверить через `PackageManager.getApplicationInfo(...)` какие из них реально установлены на устройстве, и для каждого установленного вызвать `builder.addDisallowedApplication(pkg)`. Если приложения нет — пропустить (метод бросает `NameNotFoundException`, ловим и игнорим).
3. Список передаётся в нативку через `MethodChannel` при каждом запуске VPN.

#### Уровень 2: Routing rules sing-box (DNS / IP / domain bypass)

Per-app fallback покрывает только установленные приложения. Если юзер откроет российский сайт в Chrome или любом другом браузере — нужно отрубить через **routing rules sing-box**, который встроен в ядро.

Codex обязан создать sing-box JSON-конфиг, в `route.rules` которого добавлены правила:

```json
{
  "route": {
    "rules": [
      {
        "rule_set": ["geosite-category-ru", "geosite-yandex", "geosite-vk"],
        "outbound": "direct"
      },
      {
        "rule_set": ["geoip-ru"],
        "outbound": "direct"
      },
      {
        "domain_suffix": [".ru", ".рф", ".su", ".tatar", ".moscow"],
        "outbound": "direct"
      },
      {
        "domain_keyword": ["yandex", "sberbank", "tinkoff", "vtb", "alfabank", "gazprombank", "wildberries", "ozon", "avito", "kinopoisk", "rutube", "vk.com", "mail.ru", "ok.ru", "gosuslugi", "mos.ru", "rzd.ru"],
        "outbound": "direct"
      }
    ],
    "rule_set": [
      {
        "tag": "geosite-category-ru",
        "type": "remote",
        "format": "binary",
        "url": "https://raw.githubusercontent.com/SagerNet/sing-geosite/rule-set/geosite-category-ru.srs",
        "download_detour": "direct"
      },
      {
        "tag": "geosite-yandex",
        "type": "remote",
        "format": "binary",
        "url": "https://raw.githubusercontent.com/SagerNet/sing-geosite/rule-set/geosite-yandex.srs",
        "download_detour": "direct"
      },
      {
        "tag": "geosite-vk",
        "type": "remote",
        "format": "binary",
        "url": "https://raw.githubusercontent.com/SagerNet/sing-geosite/rule-set/geosite-vk.srs",
        "download_detour": "direct"
      },
      {
        "tag": "geoip-ru",
        "type": "remote",
        "format": "binary",
        "url": "https://raw.githubusercontent.com/SagerNet/sing-geoip/rule-set/geoip-ru.srs",
        "download_detour": "direct"
      }
    ]
  }

```

**Что это даёт:**
- `geosite-category-ru` — официальный список доменов российских сервисов от мейнтейнеров sing-box, обновляется регулярно. Покрывает тысячи сайтов.
- `geosite-yandex` и `geosite-vk` — отдельные точные списки, на случай если основной не покрыл.
- `geoip-ru` — все IP-блоки, зарегистрированные в RIPE как российские. Если запрос идёт по голому IP — тоже завернётся в direct.
- `domain_suffix` — домены первого уровня `.ru`, `.рф`, `.su`, `.tatar`, `.moscow`.
- `domain_keyword` — fallback по ключевым словам в домене на случай если правила выше не сработали.
- Все эти правила имеют `outbound: "direct"` — трафик идёт мимо VPN, напрямую через домашний интернет.
- `rule_set` ресурсы кешируются локально и автообновляются раз в неделю.

**Поведение в UI:** в первой версии переключатель «Обход российских сервисов» в настройках **включён по умолчанию и заблокирован**. Юзер не может его отключить. Объяснение в подсказке: «Российские сайты и приложения работают напрямую, без VPN. Это сделано для совместимости с банками и сервисами.» Во второй версии можно дать юзеру возможность отключать, если будет запрос.

### 7.10. Соответствие политикам Google Play Store и Apple App Store

**Это критично.** Любое нарушение приведёт к удалению приложения из стора, блокировке аккаунта разработчика, потере денег за регистрацию ($25 + $99). Codex обязан следить за этим на каждом шаге.

#### 7.10.1. Категория VPN — особые требования Google Play

Приложения VPN относятся к **специальной категории** в Google Play, для них действуют дополнительные правила:

1. **Только `VpnService` API.** Приложение должно использовать **исключительно** Android `VpnService` для туннелирования. Никаких системных хаков, никакого root, никакого `iptables`-фокусничества. У нас sing-box работает через `VpnService` — это OK.

2. **Декларация в Play Console.** При публикации в Google Play Console надо явно указать в секции **App content → VPN service**, что приложение использует `VpnService`, и описать **зачем**: «обеспечение безопасного и приватного доступа к интернету для пользователей в Таджикистане». Это обязательное поле.

3. **Запрет на сбор данных без согласия.** Запрещено собирать данные о трафике юзеров, их посещениях, IP-адресах, устройствах — без явного согласия. В нашем приложении мы **ничего не собираем**: нет аналитики, нет Firebase Analytics, нет Crashlytics, нет третьих SDK с трекерами. Это сильное преимущество — одобрение пройдёт быстрее.

4. **Запрет на использование `VpnService` для нелегитимных целей.** Google запрещает использовать VpnService для блокировки рекламы в чужих приложениях, для модификации трафика других приложений, для перехвата данных. Мы не делаем ничего из этого — просто туннелируем.

5. **Возрастной рейтинг.** VPN-приложения обычно получают рейтинг 13+ или 17+. Заполняем форму честно.

6. **Privacy Policy ОБЯЗАТЕЛЬНА.** На `vpnjon.com/privacy` должна быть размещена страница с политикой конфиденциальности. Без неё приложение **не пройдёт ревью**. Содержание:
   - Что мы НЕ собираем (имена, email, телефоны, IP, историю посещений, логи трафика).
   - Что мы храним локально (выбранный язык, выбранный сервер, кеш списка серверов).
   - Что мы отправляем на наш сервер (только запрос за списком серверов, без идентификации юзера).
   - Контакты для связи (email + Telegram).
   - Дата последнего обновления.
   - Версии на русском и таджикском.
   - Codex должен сгенерировать готовый HTML-файл `privacy.html` в Фазе 11, я просто залью на сервер.

7. **Terms of Service** — желательно, не обязательно, но добавим: `vpnjon.com/terms`.

#### 7.10.2. Запреты Google Play (что НЕЛЬЗЯ ни в коем случае)

- ❌ **Не использовать запрещённые SDK.** Никаких SDK от российских компаний (Yandex Metrica, AppMetrica, VK SDK, MyTracker, MyTarget) — они в чёрном списке Google и автоматически валят приложение на ревью.
- ❌ **Не использовать заимствованные иконки/логотипы/тексты** других VPN-сервисов. Всё своё.
- ❌ **Не использовать слова в названии и описании, связанные с обходом блокировок** в явном виде. Допустимо: «защищённое соединение», «приватный доступ», «безопасный интернет». Запрещено: «обход блокировок Роскомнадзора», «обход цензуры», «заблокированные сайты». Google ревьюеры за такие формулировки заворачивают.
- ❌ **Не упоминать конкретные заблокированные ресурсы** (Instagram, Facebook, LinkedIn) в листинге Play Store. Можно сказать «доступ ко всему контенту в интернете».
- ❌ **Не давать обещаний 100% анонимности и абсолютной безопасности.** Это юридически опасно. Формулировка: «помогает защитить ваше соединение».
- ❌ **Никаких бэкдоров, скрытого функционала, скрытого включения камеры/микрофона, чтения SMS, доступа к контактам.** У нашего приложения **минимальный** список разрешений: только `INTERNET`, `ACCESS_NETWORK_STATE`, `FOREGROUND_SERVICE`. Никаких `READ_CONTACTS`, `READ_SMS`, `CAMERA`, `RECORD_AUDIO`. Codex должен следить, чтобы случайно не подтянулся пакет, требующий лишних permission'ов.
- ❌ **Никаких тёмных паттернов:** автоподписки без согласия, кнопки «Продолжить», ведущие к платежу, обманчивые модалки. У нас приложение бесплатное и не имеет покупок — это не наша проблема, но если когда-то добавим — будет.
- ❌ **Обфускация кода допустима**, скрытие функционала от ревьюеров — нет. Если Google попросит исходники — даём.

#### 7.10.3. Apple App Store — дополнительные требования (для Фазы 12)

- App Store ещё строже Google Play. VPN-приложения обязательно должны использовать **`NEPacketTunnelProvider`** через NetworkExtension framework. У нас это будет сделано через интеграцию sing-box на iOS.
- В App Store Connect при подаче надо явно указать, что приложение использует VPN и зачем.
- Privacy Nutrition Labels (App Privacy) — обязательно, заполняем «Data Not Collected».
- Apple запрещает использование `UIWebView` (deprecated) — используем только `WKWebView`. Во Flutter это автоматом.
- Запрещено собирать IDFA без явного запроса через `App Tracking Transparency`. Мы не собираем — пропускаем.
- Для VPN-приложений обязательна декларация в Info.plist:
  ```xml
  <key>NSLocalNetworkUsageDescription</key>
  <string>VPNjon использует локальную сеть для управления VPN-соединением.</string>
  ```

### 7.11. Защита ключей в приложении

**Уровни (применяем все):**

1. **Транспорт:** HTTPS обязательно, плюс **Certificate Pinning** — приложение принимает соединение к `api.vpnjon.com` только если SHA-256 fingerprint сертификата совпадает с зашитым. Реализация через пакет `dio` + `dio_certificate_pinning` или вручную через `HttpClient.badCertificateCallback`.

2. **Подпись запросов:** HMAC-SHA256 параметр `t`, вычисляется в нативном коде через FFI / NDK, секрет лежит в скомпилированной `.so`-библиотеке, не в Dart. Это не идеальная защита, но на порядок усложняет реверс по сравнению с константой в Dart-коде.

3. **Шифрование payload:** ответ сервера — AES-256-GCM, ключ шифрования также в нативной библиотеке.

4. **In-memory only:** расшифрованный JSON живёт только в RAM (Riverpod state), на диск пишется уже **повторно зашифрованным** через `flutter_secure_storage` (использует Android Keystore).

5. **UI-уровень:** в приложении **нигде** нельзя увидеть, скопировать, экспортировать, расшарить VLESS-ключ. Никаких `Share`, никаких «показать ключ», никакого debug screen.

6. **Code obfuscation:** сборка релизного APK через `flutter build apk --release --obfuscate --split-debug-info=build/symbols`. Это переименовывает все Dart-символы.

7. **R8/ProGuard:** включён для нативного Android-кода, переименовывает классы Java/Kotlin.

8. **Anti-debug (опционально, на потом):** проверка `Debug.isDebuggerConnected()` на старте, в проде блокирует запуск. Не критично для первой версии.

9. **Ротация ключей:** через команду бота `/rotate_keys` раз в неделю или по требованию. Это убивает мотивацию вытаскивать ключи — даже если кто-то их выдрал, они мёртвые через неделю.

**Чего НЕ делаем (избыточно для нашего масштаба):**
- Google Play Integrity API (отвалятся юзеры с кастомными прошивками).
- Серверная attestation устройств.
- Динамическая загрузка кода через JNI dlopen.
